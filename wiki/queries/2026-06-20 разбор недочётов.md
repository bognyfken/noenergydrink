# Разбор недочётов проекта (этап 2)

**Last updated:** 2026-06-20
**Sources:** ревью кода + проверка живого dev-сервера (Playwright, чистая загрузка)
**Контекст:** ответ на вопрос «какие недочёты на данном этапе». Ядро трекера
написано аккуратно; слабые места — по краям (обработка ошибок, приватность кода,
несведённый визуал). На чистой загрузке приложение стартует без ошибок в консоли.

---

## 🔴 Критичное

### 1. Нет Error Boundary → любой сбой = белый экран
Если экран бросает исключение, React размонтирует всё дерево и остаётся пустота
без сообщения. Это и есть наблюдавшееся «ничего не открывается». Нужен
`<ErrorBoundary>` вокруг роутов с человеческим фолбэком («Что-то сломалось,
перезагрузи»). Дёшево, маскирует целый класс сбоев.

### 2. Латентный баг жизненного цикла БД
`DatabaseClosedError: Maximum call stack size exceeded` (был в логе dev-сервера).
Причина — `openDb()` делает `active?.close()`; если в этот момент летит
незавершённый `toArray()` из предыдущего `hydrate`, соединение рвётся под
транзакцией. На чистой загрузке **не воспроизводится** (транзиент от HMR +
перезапусков по `.env` + нескольких подключений к одной IndexedDB). Риск
остаётся под HMR / повторным `init()` / двумя вкладками. Чинить: не закрывать
базу при том же имени; не дёргать `init()` повторно. См. [[Architecture]].

## 🟠 Безопасность / приватность (важно из-за мед-контекста)

### 3. Имена профилей публичны (осознанно, но стоит помнить)
`cloudListProfiles()` отдаёт `name, salt, verifier` любому с anon-ключом (ключ в
бандле). RLS включён, но политика под `anon` это разрешает намеренно (нужно для
экрана выбора профиля). Следствие — список имён виден, а `salt/verifier` дают
материал для оффлайн-перебора кода.

### 4. Короткий код перебираем оффлайн
`MIN_CODE_LENGTH = 4` + `inputMode="numeric"` → 10⁴ комбинаций. `salt` и `id`
публичны, `did` выводится из `id+salt+code` (`auth.ts: deriveDid`) — единственный
секрет это код. 4 цифры перебираются мгновенно → восстанавливается `did` →
читаются все дни и заметки. Рекомендация: минимум 6+ символов, подсказывать
буквенный код. Радикально — Supabase Auth (есть в [[Roadmap]] как «позже»).

### 5. RLS — включён, перепроверять при изменении схемы
По логу RLS на всех 4 таблицах работает. Замечание-напоминание: при любой правке
таблиц/политик убедиться, что `entries`/`achievements` не читаются/не пишутся
по чужому `user_id` через anon-ключ. См. [[Data Model]].

## 🟡 Надёжность данных

### 6. Оффлайн-правки не дослыаются до следующего hydrate
`setDay`/`setNote` пушат в облако через `void cloudUpsertEntry()` без ожидания и
без очереди ретраев. При обрыве сети правка остаётся локально и долетает лишь при
следующем `hydrate()` (там есть back-fill — спасает). Для одного устройства ок;
при двух девайсах окно рассинхрона реальное. Решение — мелкая outbox-очередь.

## 🟣 Визуал / UX

### 7. Новые медали не подключены
`AchievementsScreen` рисует эмодзи 🔥 и `Lock`, хотя в `public/achievements/d*.webp`
уже лежат медали под каждую веху (d1…d100, нарезаны из ассета Gemini). Картинки
готовы, но не используются. См. [[Design]].

### 8. Логотип не используется на входе
В `AuthScreen` сверху дефолтный `Flame` из lucide. Есть фирменный `public/logo.png`
(банка в сердце) — логичнее показать его.

### 9. Онбординга нет
Профиль создаётся только с именем и кодом. Негде ввести дату старта воздержания,
тон ассистента, аватар (обсуждалось как онбординг/настройки). См. [[Roadmap]].

## ⚪ Мелочи
- `public/icons/icon-512.png` — 326 КБ (PNG с градиентом); pngquant ужал бы до ~80 КБ.
- Рассогласование: подсказка кода говорит «любой длины — например, важная дата»,
  но `inputMode="numeric"` зовёт цифровую клавиатуру (буквы вводить неудобно).
- Тесты только на `streak`. `mergeByUpdatedAt`/`hydrate` (самое конфликтное) не покрыты.
